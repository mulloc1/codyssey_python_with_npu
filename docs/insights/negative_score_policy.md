# 음수 점수 정책 메모

## 배경

`judge_cross_vs_x()`는 현재 점수의 대소 비교와 epsilon 동점 규칙으로 판정한다.

- `abs(score_cross - score_x) < epsilon` -> `UNDECIDED`
- 그 외 `score_cross > score_x` -> `Cross`, 아니면 `X`

이 로직은 음수 점수도 기술적으로 처리 가능하지만, **정책 의미**를 어떻게 둘지 의사결정이 필요하다.

## 핵심 쟁점

음수를 허용하면 판정은 "절대적으로 얼마나 유사한가"보다 "둘 중 상대적으로 누가 더 높은가" 성격이 강해진다.

예:
- `score_cross = -2`, `score_x = -5` -> `Cross`
- 해석: 둘 다 불일치 경향이지만 상대적으로 덜 나쁜 쪽이 `Cross`

## 선택지

### 옵션 A) 입력 범위 제한(권장, 단순)

- 정책: 입력/필터 값을 `0/1` 또는 `[0, 1]`로 제한
- 장점:
  - 판정 의미가 직관적임(높을수록 유사)
  - 테스트/설명/리포트가 단순
- 단점:
  - 표현력이 제한됨

### 옵션 B) 음수 허용 + 상대 비교 정책 명시

- 정책: 음수를 허용하되 현재 판정 규칙 유지
- 장점:
  - 유연함, 확장성 높음
- 단점:
  - "둘 다 음수인데 왜 Cross?" 같은 해석 이슈 발생 가능
  - 문서화/테스트 보강 필요
