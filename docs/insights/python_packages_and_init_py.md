# 파이썬 패키지와 `__init__.py` (점 경로 import 안정성)

`src.npu.mac`처럼 **점(`.`)으로 이어지는 import**가 어떻게 동작하는지, `__init__.py`를 두는 이유를 정리한다.

## 점 경로 import가 의미하는 것

`from src.npu.mac import compute_mac`을 실행하면, import 시스템은 대략 다음 순서로 이름을 해석한다.

1. **`src`** — 패키지(디렉터리) 또는 최상위 네임스페이스
2. **`npu`** — 그 아래의 하위 패키지
3. **`mac`** — 그 아래의 모듈(파일 `mac.py`)

각 단계가 **패키지로 인식**되어야 다음 단계로 넘어갈 수 있다. 여기서 말하는 “패키지”의 전통적인 표시가 바로 디렉터리 안의 **`__init__.py`**이다.

## `__init__.py`가 하는 일

- 디렉터리를 **일반 패키지(regular package)**로 명시한다.
- 파일이 비어 있어도 “이 폴더는 패키지다”라는 **표식** 역할을 한다.
- 내용을 채우면 **`from src.npu import …`**처럼 하위 모듈의 이름을 **한 단계에서 재노출**하거나, `__all__`로 공개 API를 정할 수 있다.  
  (`python_with_npu`의 `src/npu/__init__.py`가 이 패턴에 해당한다.)

## `__init__.py` 없이도 되는 경우가 있는가

파이썬 3.3 이후 **PEP 420(네임스페이스 패키지)** 덕분에, 조건이 맞으면 `__init__.py` 없는 디렉터리도 패키지처럼 참여할 수 있다.

다만 다음 이유로 **“없어도 된다”를 전제로 레이아웃을 짜기는 보통 권하지 않는다.**

- **상위/하위 패키지에 `__init__.py`가 있는지**, **`sys.path`에 무엇이 올라가는지**에 따라 동작이 달라질 수 있다.
- **mypy**, **IDE**, **unittest discovery** 등은 `__init__.py` 유무에 더 민감한 경우가 있다.
- 재현하기 어려운 **“내 PC에서는 되는데 CI에서는 안 됨”**류 이슈로 이어지기 쉽다.

즉, **버전 하나만의 문제가 아니라 “파이썬 버전 + 실행 방식 + 도구” 조합**에서 불안정해질 수 있다고 보는 것이 정확하다.

## 이 프로젝트에서의 실용적 결론

- **`__init__.py`를 두는 쪽이 import와 도구 호환 측면에서 보수적이고 안전하다.**
- `src/npu/__init__.py`처럼 **재노출·`__all__`**을 쓰는 경우에는, 그 파일 자체가 **공개 API를 문서화하는 역할**도 한다.
- 재노출이 필요 없다면 **`__init__.py`를 비우거나 최소한으로 유지**하는 선택은 가능하지만, **파일을 아예 없애는 것**과는 다르다. 디렉터리를 패키지로 고정하려면 **빈 `__init__.py`라도 두는 편**이 일반적이다.
