# 코드 구조·원리·심층 인터뷰 정리

과제/면접형 문항(항목 2~4)에 대해 **이 레포의 실제 코드**를 기준으로 답을 정리한 문서다.  
문항에서 말한 `normalize_label`은 본 프로젝트에서는 **`normalize_expected`**, **`normalize_filter_key`**([`src/npu/labels.py`](../../src/npu/labels.py)), JSON 조합용 **`normalize_expected_and_filter_key`**([`src/npu_io/label_normalization.py`](../../src/npu_io/label_normalization.py))로 나뉜다.

## 항목 2. 코드 구조 및 설계

### MAC 연산 로직의 전체 흐름(입력 → 위치별 곱 → 누적 합)

코드 기준 흐름은 다음 두 층으로 나뉜다.

1. **입력 검증** — [`validate_mac_inputs`](../../src/npu/mac.py): `validate_matrix`로 패턴·필터가 각각 정사각 `N×N`인지 확인하고, 둘의 `N`이 같을 때만 그 `N`을 반환한다. 순수 MAC 루프에서는 검증을 반복하지 않도록 [`compute_mac`](../../src/npu/mac.py)은 검증 없이 호출한다는 전제가 docstring에 적혀 있다.
2. **연산** — `compute_mac`: `iRow`, `iCol` 이중 루프로 `pattern[i][j] * filter[i][j]`를 누적 변수 `fAcc`에 더하고, 최종적으로 `float` 점수 하나를 반환한다.

즉 “입력(검증된 텐서) → 동일 좌표끼리 곱 → 전 좌표에 대해 합”이 [`mac.py`](../../src/npu/mac.py) 한 파일에 압축되어 있다.

### `patterns` 키에서 N 추출과 `filters.size_N` 매칭 설계

- **N 추출**: [`src/npu_io/schema.py`](../../src/npu_io/schema.py)의 `extract_size_from_pattern_key`가 정규식 `^size_(\d+)_(\d+)$`로 패턴 키(예: `size_13_002`)의 **첫 번째 숫자 그룹**을 `N`으로 파싱한다. 형식이 맞지 않으면 즉시 `ValueError`로 실패시켜 잘못된 키가 조용히 넘어가지 않게 했다.
- **필터 선택**: 같은 모듈의 `select_filters_for_size`가 `filters` 객체에서 키 문자열 `f"size_{iSize}"`로 조회한다. `patterns`에서 나온 `N`과 동일한 정수로 맞추므로, “패턴이 요구하는 크기”와 “로드할 필터 묶음”이 한 축으로 정렬된다.
- **그 다음**: `data_json_mode` 루프에서 추출한 `N`으로 위 맵을 고른 뒤, `validate_pattern_and_filters`로 `input` 행렬과 필터 행렬이 모두 `N×N`인지 한 번에 검증한다. 키 파싱·조회·형상 검증이 역할별 함수로 쪼개져 있어 테스트와 오류 메시지 해석이 쉬워진다.

### 라벨 정규화를 별도 함수·모듈로 둔 이유와 장점

이름은 문항의 `normalize_label`이 아니라 아래처럼 분리되어 있다.

| 위치 | 역할 |
|------|------|
| `src/npu/labels.py` | 도메인 표준 라벨(`Cross`/`X`)로의 매핑 규칙만 담당 |
| `src/npu_io/label_normalization.py` | JSON 로딩 맥락에서 expected·필터 키를 한 번에 정규화하는 얇은 조합 |

**이유·장점**

- **단일 책임**: MAC·판정 코드가 문자열 별칭(`+`, `cross`, 대소문자 등)을 알 필요가 없어진다. 비교는 항상 표준 라벨끼리.
- **재사용·테스트**: [`tests/npu_tests/test_labels.py`](../../tests/npu_tests/test_labels.py)처럼 순수 함수만 단위 테스트하면 된다. 콘솔·JSON 경로가 달라도 규칙은 한곳에서 수정한다.
- **경계 분리**: I/O(JSON) 조합(`normalize_expected_and_filter_key`)과 순수 도메인 맵(`normalize_expected`)을 나누면, 나중에 입력 소스가 늘어도 I/O 쪽만 확장하기 쉽다.

### 시간 측정 시 I/O를 배제하고 “연산 구간”만 보려고 잡은 경계

문항의 “VO”는 **입출력(I/O)** 을 뜻하는 것으로 이해하고 정리한다(파일 읽기, `print`, 사용자 입력 등은 타이머 밖).

- **벤치마크**([`src/npu/benchmark.py`](../../src/npu/benchmark.py)): `benchmark_mac_average`에서 `validate_mac_inputs`는 **타이머 시작 전**에 한 번 호출되고, `perf_counter` 구간 안에는 **`compute_mac`만** 들어간다. 반복마다 검증 비용을 섞지 않아 “MAC 본체”에 가까운 시간만 평균 낸다.
- **콘솔·JSON 모드의 한 줄 시간**([`user_input_3x3.py`](../../src/app/user_input_3x3.py), [`data_json_mode.py`](../../src/app/data_json_mode.py)): `fStart`~`fEnd`는 **두 번의 `compute_mac`(A/B 또는 Cross/X)** 만 포함한다. 그 앞의 `validate_mac_inputs`, 그 뒤의 `judge_ab` / `judge_cross_vs_x`는 구간 밖이다. 따라서 출력되는 ms는 “판정까지 전부”가 아니라 **MAC 연산 위주**에 가깝다.

완전히 순수 연산만 남기려면 벤치 쪽 정의(위와 같음)를 기준으로 설명하는 것이 맞다.

## 항목 3. 핵심 기술 원리 적용

### “MAC 점수가 높을수록 더 유사하다”고 말할 수 있는 이유

같은 크기의 패턴 `P`와 필터 `F`에 대해 MAC은 \(\sum_{i,j} P_{ij} F_{ij}\) 즉 **내적에 가까운 스칼라**다. 같은 위치에서 둘 다 크고 같은 부호면 항이 커지고, 한쪽이 0이면 기여가 없다.  
고정된 두 필터 중 “어느 쪽과 더 비슷한 형태인가”를 **같은 패턴으로 각각 한 번씩 MAC**한 뒤 점수가 더 큰 쪽을 고르면, “그 필터 템플릿과 방향이 맞는 활성이 많이 겹친다”는 해석이 가능하다. (스케일·정규화 없이는 절대값만으로는 한계가 있어, 이 프로젝트는 **동일 패턴에 대한 상대 비교**에 초점을 둔다.)

### 패턴 크기 N일 때 연산 횟수 N²·측정이 O(N²) 경향을 보이는 이유

이중 루프가 `range(iSize)` 두 겹이라 **곱·덧셈 연산은 대략 N²회**다. 벤치마크 표의 `연산 횟수(N²)`도 이에 맞춘다.  
이론적 시간 복잡도는 입력 크기에 대해 **Θ(N²)** 이고, Python 구현에서는 루프·인덱싱 오버헤드가 붙지만, N을 키울수록 지배항이 N²에 비례해 커지므로 측정 곡선이 **N² 증가와 같은 방향**으로 가는 것이 자연스럽다.

### 부동소수점 오차와 epsilon 비교가 필요한 이유

#### 왜 오차가 생기는가(원리)

Python의 `float`는 대부분의 환경에서 **IEEE 754 배정밀도(binary64)** 를 쓴다. 실수 전체가 아니라 **부호·지수·가수(유효숫자 비트)로 표현 가능한 유한 개의 이진 유리수**만 정확히 담을 수 있다.

1. **표현 불가능한 값**  
   십진수 `0.1`처럼 이진 소수로 **유한 자리로 쓸 수 없는** 수는, 저장 시 **가장 가까운 표현 가능한 비트 패턴으로 반올림**된다. 그 순간 이미 “진짜 0.1”과 미세한 괴리가 생긴다.

2. **연산마다 다시 반올림**  
   곱셈·덧셈 결과의 정확한 비트 수는 가수보다 길어질 수 있어, 하드웨어/언어 규격에 따라 **한 번의 연산마다 정밀도 안으로 잘라 내는 반올림**이 일어난다. MAC처럼 `곱 → 누적`을 N²번 반복하면, 매 단계의 작은 오차가 **누적 합 `fAcc`에 차곡차곡 섞인다**.

3. **같은 수학, 다른 구현 순서**  
   결합법칙이 부동소수점에서는 성립하지 않는다. 항 순서를 바꾸면 누적 오차 패턴이 달라져, 이론적으로 같은 합이라도 마지막 비트가 어긋날 수 있다.

4. **큰 수와 작은 수를 같이 더할 때**  
   지수 차이가 크면 작은 쪽의 유효숫자 일부가 **가수 비트 한도 때문에 떨어져 나가** 사실상 무시되는 현상(대소 비교·누적에서 “큰 쪽만 남는” 효과)이 생길 수 있다. MAC에서 극단적인 스케일 혼합이 있으면 같은 이슈가 이론적으로 가능하다.

정리하면, 오차는 “버그”라기보다 **유한 비트로 실수를 근사하고, 매 연산 후 규격에 맞게 반올림하는 구조의 필연적 결과**다.

#### 이 프로젝트에서 epsilon이 하는 일

수학적으로 두 MAC 점수가 같아도, 위 이유로 저장된 `float`는 `a == b`가 거짓일 수 있다. 반대로 **아주 조금만 다른데 동률로 취급해야 할 때**도 임계값이 없으면 정책이 흔들린다.  
그래서 `abs(a - b) < ε`로 **동률대역**을 두고, 그 안이면 `UNDECIDED`로 묶는다. 본 프로젝트는 [`DEFAULT_EPSILON`](../../src/npu/constants.py)과 [`judge_ab` / `judge_cross_vs_x`](../../src/npu/judgement.py)에서 이를 공유한다. ε는 “항상 옳은 절대 기준”이 아니라, **데이터 스케일과 허용 동률 범위를 설계로 고정한 값**으로 이해하는 것이 좋다.

### 동점 처리를 모드 1 vs 모드 2에서 다르게 보이는 이유(요구사항 관점)

- **판정 규칙 자체**는 동일하다: `abs(차이) < ε`이면 내부적으로 `UNDECIDED`([`LABEL_UNDECIDED`](../../src/npu/constants.py)).
- **모드 1(사용자 입력)**: 사람이 읽기 쉽게 **`판정 불가`**라는 한글 표기로 출력한다. 승자가 없다는 UX에 맞춘다.
- **모드 2(JSON)**: 기계가 읽는 **`UNDECIDED`** 문자열을 그대로 쓰고, PASS는 `verdict == expected`일 때만 주므로 기대가 `Cross`/`X`이면 동률은 **FAIL로 집계**된다. 자동 채점·데이터 검증에서는 “명확한 승자와 기대가 일치했는지”를 보려는 요구와 맞다.

## 항목 4. 심층 인터뷰

### `data.json`에 새 라벨(예: 또 다른 패턴 클래스)이 추가된다면 확장 순서

문항 예시가 `'0)`처럼 깨져 있으면, **새 기호 하나가 새 클래스를 뜻한다**고 가정해 순서만 정리한다.

1. **도메인 상수**: [`src/npu/constants.py`](../../src/npu/constants.py)에 표준 문자열 라벨 추가.
2. **정규화**: [`src/npu/labels.py`](../../src/npu/labels.py)의 맵에 raw 문자열 → 표준 라벨. JSON 조합이 필요하면 [`label_normalization.py`](../../src/npu_io/label_normalization.py) 시그니처·호출부 정리.
3. **판정**: 지금은 이진 비교(`judge_cross_vs_x`) 또는 A/B(`judge_ab`)다. 새 라벨이 **세 번째 후보와 점수 비교**를 요구하면 판정 함수를 다중 비교나 argmax 형태로 확장하고, epsilon·동률 규칙을 명문화한다.
4. **데이터 스키마**: `filters`에 해당 크기의 새 키 행렬이 필요한지, `patterns.expected` 허용 값 목록을 README·스키마 검증에 반영.
5. **출력·스트**: `data_json_mode`의 한 줄 포맷, `summarize_results` 실패 사유 문자열, 필요 시 리포트 섹션 문구.
6. **테스트**: `test_labels`, `test_judgement`, JSON 통합 경로에 케이스 추가.

### 1000×1000 패턴일 때 시간·메모리와 우선 최적화

- **메모리**: 패턴 1개 + 필터 2개만 해도 float 64비트 기준 대략 **10⁶ × 3 × 8바이트 ≈ 24MB** 수준(실제는 리스트 오버헤드로 더 큼). 여러 케이스·중간 버퍼가 있으면 금방 기가바이트 근처로도 갈 수 있다.
- **시간**: 순수 Python 이중 루프는 N²이지만 상수항이 커서 1000×1000도 체감될 수 있고, 반복 벤치·대량 케이스에서는 병목이 된다.

**우선 적용 후보**

1. **NumPy 등으로 벡터화**된 내적/원소곱 합으로 같은 수학을 유지하면서 C 레벨로 내린다.
2. **dtype을 float32**로 낮춰 메모리·대역폭을 절반 가까이 줄인다(정밀도 트레이드오프).
3. **블록/타일링**으로 캐시 친화적 접근(극단적 N에서).
4. I/O와 분리된 **미세 벤치**는 유지해, 최적화 전후 동일 경계로 비교한다.

### CPU 직렬 처리 vs NPU 병렬 관점에서 이 MAC 문제의 병목

- **현재 Python 구현**: 스칼라 루프가 CPU에서 **직렬**로 돈다. N²이 커지면 **연산 개수 자체**와 **파이썬 인터프리터/객체 접근 비용**이 병목이 된다.
- **NPU(및 GPU) 관점**: MAC은 **서로 다른 (i,j)에 대한 곱·덧셈을 독립**에 가깝게 수행할 수 있어 **많은 MAC 유닛에 분산**하기 좋다. 이때 병목은 종종 **메모리 대역폭**(패턴·필터를 각 유닛에 공급하는 속도)이나 **동기화·데이터 레이아웃**으로 옮겨간다.

즉 같은 수식이라도, 직렬 CPU 코드는 “루프+언어 오버헤드”, 병렬 가속기는 “처리 유닛 수 대비 메모리 공급”이 각각 한계축이 된다고 설명할 수 있다.
