# Mini NPU 과제 구현 체크리스트 (작업 순서 + 커밋 단위 + 테스트 구조)

이 문서는 `docs/subject.md` 요구사항을 **실행 순서**로 분해한 체크리스트다.  
기준: `docs/commit_guidelines.md`의 원칙(한 커밋 = 한 논리적 변경, 요구사항 경계 기준 분리).

`docs/testing_guidelines.md`의 **폴더 미러링·네이밍·단일 책임·외부 의존 최소화** 규칙을 각 단계에 맞춰 아래에 적용한다.

---

## 사용 방법

- 아래 단계는 기본적으로 **위에서 아래 순서대로 진행**한다.
- 예외적으로 **패턴 생성기(10-2)는 Step 2(MAC 코어) 직후 조기 적용**을 권장한다. 이후 테스트/벤치마크 입력 생성을 자동화할 수 있어 개발 속도가 빨라진다.
- 각 단계는 가능한 한 **1커밋 1주제**로 마무리한다.
- 체크박스는 구현 + **해당 단계에 적은 단위 테스트(또는 스모크)** 까지 끝났을 때 체크한다.

---

## 테스트 배치 공통 원칙 (`docs/testing_guidelines.md`)

- **폴더 미러링(최우선)**: `src/<path>/<module>.py` → `tests/<path>/test_<module>.py`
- **파일명**: `test_*.py`, 테스트 함수/클래스 이름은 **행동이 드러나게**
- **단일 책임**: 한 테스트는 한 가지 실패 원인만 드러나게 `assert`를 좁게
- **`print` 사용 원칙**: 합격/실패는 `assert`가 판정하고, `print`는 필요 시 실패 맥락 보조용으로만 사용
- **외부 의존 최소화**: 네트워크 금지, **시간 측정은 가능하면 모킹**하거나 “반복 횟수 1회·고정 시드”로 단순화한 테스트만 유닛에 둠
- **파일 I/O**: `data.json` 읽기 로직은 **`tempfile`로 만든 임시 JSON**으로 검증
- **실행**:

```bash
python3 -m unittest discover -s tests -p 'test_*.py' -v
```

아래 단계별 **「유닛 분리·테스트」** 는 저장소에 `src/`가 아직 비어 있어도, **처음부터 이 구조로 파일을 만들 것**을 전제로 한 권장안이다. 실제 모듈명은 팀/개인 취향에 맞게 조정해도 되지만, **`src`와 `tests` 경로만 미러링**하면 된다.

## 0) 초기 세팅

- [ ] 프로젝트 실행 진입점(`main.py`)과 기본 폴더 구조를 확인한다.
- [ ] `docs/subject.md`, `docs/testing_guidelines.md`, `docs/commit_guidelines.md`를 읽고 범위를 확정한다.
- [ ] 커밋 메시지 규칙(`feat/fix/docs/test/refactor/chore`)을 먼저 정한다.
- [ ] `src/` 패키지(예: `src/npu/`, `src/io/` 등)와 `tests/` 루트를 만든다.

### 유닛 분리·테스트

- **구조**: `tests/__init__.py`(필요 시), `src/__init__.py` 등 패키지 초기화만 두어도 됨.
- **테스트**: 이 단계에서는 **스모크 1개**만 권장 — `import` 가능 여부 또는 `python3 -m unittest discover`가 빈 상태에서도 실패하지 않게 정리.
- **커밋 분리**: 코드 없이 디렉터리만이면 `chore`, 테스트 뼈대 추가면 `test:` 또는 `chore:`.

권장 커밋 예시:

```text
chore: Mini NPU 과제 기본 실행 구조 점검
```

## 3) 점수 비교/판정 정책

목표: epsilon 기반으로 Cross/X/UNDECIDED를 판정한다.

- [ ] `abs(score_a - score_b) < epsilon` 동점 정책을 적용한다. 기본값 `epsilon = 1e-9`를 상수로 정의한다.
- [ ] 판정 결과를 `Cross`, `X`, `UNDECIDED` 중 하나로 반환한다.
- [ ] 판정 함수가 입력 모드/JSON 모드에서 공통으로 재사용되게 한다.

완료 기준:

- [ ] 동점 케이스에서 `UNDECIDED`가 안정적으로 나온다.
- [ ] 비동점 케이스에서 점수 큰 쪽으로 일관되게 판정된다.

### 유닛 분리·테스트

| 구분 | 권장 경로 |
|------|-----------|
| 구현 | 예: `src/npu/judgement.py` — `judge_cross_vs_x(score_cross, score_x, epsilon=1e-9)` |
| 테스트 | `tests/npu_tests/test_judgement.py` |
| 검증 예시 | 동점(`|a-b| < eps`), Cross 우세, X 우세, 경계값(`|a-b| == eps`는 설계한 규칙대로 한 가지만 명시) |

- **원칙**: 판정만 단위 테스트하고, 콘솔/JSON은 이 함수를 **호출만** 하게 한다.

권장 커밋 예시:

```text
feat: epsilon 기반 판정 로직 구현
```

## 4) 모드 1: 사용자 입력(3×3)

목표: 콘솔 입력으로 필터 A/B + 패턴을 받아 바로 판정한다.

- [ ] 프로그램 시작 시 콘솔 메뉴에서 입력 방식 선택지를 제시한다.
  - `1) 사용자 입력(3×3)`
  - `2) data.json 분석`
  - `3) 패턴 자동 생성기(보너스 연계)`
- [ ] 유효하지 않은 메뉴 입력(범위 밖/숫자 아님) 시 안내 후 재선택을 유도한다.
- [ ] 3×3 필터 2개(A/B) 입력 루프를 구현한다.
- [ ] 필터 입력 완료 후 저장 확인 메시지를 출력한다(`subject.md` Section 8 실행 흐름 기준).
- [ ] 3×3 패턴 입력 루프를 구현한다.
- [ ] 행/열 수 불일치, 숫자 파싱 실패 시 재입력 안내를 구현한다.
- [ ] 점수, 판정 결과, 연산 시간(ms)을 출력한다.

완료 기준:

- [ ] 정상 입력 시 결과가 출력된다.
- [ ] 메뉴 선택에 따라 올바른 모드(사용자 입력/JSON/자동 생성)로 분기한다.
- [ ] 잘못된 입력 시 프로그램이 종료되지 않고 재입력을 유도한다.

### 유닛 분리·테스트

| 구분 | 권장 경로 |
|------|-----------|
| 구현 | 예: `src/io/parse.py` — `parse_row(line, expected_count)`, `read_matrix_3x3_lines(lines)` 등 **문자열/리스트만** 다루는 함수 |
| 구현 | 예: `src/app/console_flow.py` — `input()` 루프는 얇게, 검증은 `parse`로 위임 |
| 테스트 | `tests/io/test_parse.py` — 잘못된 줄, 공백 여러 개, 음수/실수 등 |
| 테스트 | 콘솔 통합은 최소화하고, **순수 파서만** 유닛 테스트 |

- **원칙**: `input()`을 직접 호출하는 코드는 유닛 테스트하기 어렵다. **한 줄 파싱·행 개수 검사**를 분리해 테스트한다.

권장 커밋 예시:

```text
feat: 3×3 콘솔 입력 모드 및 입력 검증 구현
```

## 5) 라벨 정규화

목표: 외부 표기를 내부 표준 라벨로 통일한다.

> **선행 조건**: 이 단계를 완료한 뒤 Step 6(data.json 로드)을 진행한다.  
> JSON 로드 시 `expected`와 filter 키를 즉시 정규화해야 PASS/FAIL 비교가 올바르게 동작하기 때문이다.

- [ ] expected 값 `'+'`, `'x'`를 `Cross`, `X`로 정규화한다.
- [ ] filter 키 `'cross'`, `'x'`를 `Cross`, `X`로 정규화한다.
- [ ] 콘솔 출력/PASS-FAIL 비교를 표준 라벨 기준으로 통일한다.

완료 기준:

- [ ] 표기 차이 때문에 오판정되는 케이스가 없어야 한다.

### 유닛 분리·테스트

| 구분 | 권장 경로 |
|------|-----------|
| 구현 | 예: `src/npu/labels.py` — `normalize_expected()`, `normalize_filter_key()` |
| 테스트 | `tests/npu_tests/test_labels.py` — 대소문자/공백이 과제 범위에 없다면 다루지 않음(과제에 있는 매핑만) |

- **원칙**: 매핑 테이블은 **데이터 한 곳**에만 두고 테스트로 고정한다.

권장 커밋 예시:

```text
feat: 라벨 정규화 규칙 적용
```

## 6) 모드 2: `data.json` 로드/검증

목표: JSON 데이터를 일괄 판정하고 PASS/FAIL을 출력한다.

> **선행 조건**: Step 5(라벨 정규화)가 완료된 상태여야 한다.  
> 로드 직후 `expected`와 filter 키에 정규화 함수를 적용해야 이후 판정이 올바르게 동작한다.

- [ ] `filters.size_5/size_13/size_25`를 로드한다.
- [ ] `patterns.size_{N}_{idx}`에서 N을 추출해 대응 필터를 선택한다.
- [ ] 로드 직후 `expected` 값과 filter 키에 Step 5의 정규화 함수를 적용한다.
- [ ] 패턴/필터 크기 불일치 시 FAIL + 원인 메시지로 처리한다.
- [ ] 오류 케이스가 있어도 전체 실행은 계속되게 한다.

완료 기준:

- [ ] 모든 패턴 케이스에 대해 판정 + PASS/FAIL이 출력된다.
- [ ] 데이터가 일부 깨져도 비정상 종료 없이 요약까지 진행된다.

### 유닛 분리·테스트

| 구분 | 권장 경로 |
|------|-----------|
| 구현 | 예: `src/io/json_loader.py` — `load_json(path)`, `iter_pattern_cases(data)` |
| 구현 | 예: `src/io/schema.py` — 키에서 `N` 추출, 필터 선택, 크기 검증 → **결과 객체 또는 `(ok, err)`** |
| 테스트 | `tests/io/test_schema.py` — 키 파싱, 크기 불일치 |
| 테스트 | `tests/io/test_json_loader.py` — `tempfile.NamedTemporaryFile`에 JSON 쓰고 로드 |

- **원칙**: 실제 `data.json` 경로에 의존하지 않고, **임시 파일**로 최소 스키마를 검증한다.

권장 커밋 예시:

```text
feat: data.json 일괄 판정 모드 및 스키마 검증 구현
```

- [ ] 표에 `크기`, `평균 시간(ms)`, `연산 횟수(N²)`를 포함한다.

완료 기준:

- [ ] 사용자 입력 모드/JSON 모드에서 요구된 성능 출력이 나온다.

### 유닛 분리·테스트

| 구분 | 권장 경로 |
|------|-----------|

권장 커밋 예시:

